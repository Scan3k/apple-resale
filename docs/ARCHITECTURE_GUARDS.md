# ARCHITECTURE_GUARDS

## Главный принцип
Проект строится с нуля, но с учетом прошлых ошибок.  
Главный приоритет — не скорость любой ценой, а стабильная и понятная архитектура.

## Общие правила
- Не смешивать разные зоны проекта в одной задаче без необходимости
- Не делать хаотичных изменений сразу в нескольких слоях
- Не переписывать архитектуру ради локального bugfix
- Любые важные изменения сначала оценивать на уровне общей структуры проекта

## Критичные зоны
Следующие части проекта считаются критичными и не должны меняться без отдельного решения:
- модель товара Product
- общие типы
- data flow товара
- shared helper logic
- API-контракты
- структура данных каталога

## Правила по Product
- У товара должен быть единый контракт
- Нельзя плодить локальные версии типа Product в разных местах
- Изменение структуры Product считается архитектурным изменением

## Правила по слоям
- Публичная часть не должна ломать admin-логику
- Admin не должна ломать публичную часть
- Общая логика должна быть вынесена в отдельные shared/core-слои
- UI-исправления должны по возможности оставаться локальными

## Правила разработки
- Сначала проектируем, потом реализуем
- Сначала фиксируем структуру, потом добавляем функциональность
- Любая спорная задача должна сначала проходить через главный координационный чат

## Цель
Сделать проект удобным для параллельной разработки в нескольких чатах без потери целостности.
